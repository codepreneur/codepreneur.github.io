      How Public Key Cryptography works

      The simplest possible example of cryptography would look something like this:

      if you have a string, lets say ‘apple’ and you are using 2 as a ‘key’ an encrypted string of ‘apple’ would look like this: ‘crrng’. That is, all letters are moved/shifted by 2 positions in the alphabet. Now, obviously real encryption is a lot more complex than that, but the principle itself is simple.

      Let’s say for a moment that you are a customer of Barclays and you are trying to login to your online banking interface. The whole process would be something like this:

      1. Barclays generates a set of two keys: public and private.
        Public key is public. That is, everyone can see it.
        Private key is never shared. Barclays is the only company that knows it.

      2. Your browser requests the public key.

      3. Your browser generates a random encryption key.
      4. Your browser encrypts all communication using this new encryption key.
      5. Your browser encrypts the encryption key (#3) itself with Barclays public key.
      6. And sends all of that to Barclays over an insecure network.
      7. Barclays uses its private key to decrypt the encryption key.
      8. Finally, Barclays uses that encryption key to decrypt the message.

      Private and public keys make up one key and can’t be generated at different times. I.e. they have to be generated together. And keys that are generated are symmetric, because that’s less computationally expensive.

      Symmetric means it can be used for both decryption and encryption.
